# Review 检查清单

## 功能完整性

- [ ] 原有功能是否完整保留？
- [ ] 删除的功能是否经过人工确认？
- [ ] 新增功能是否有对应测试？
- [ ] API 变更是否向后兼容？

## 测试覆盖

- [ ] 所有现有测试是否通过？
- [ ] 是否覆盖边界条件？
- [ ] 是否覆盖降级场景？
- [ ] 是否覆盖错误处理？
- [ ] 测试描述是否清晰（中文）？

## 代码风格

- [ ] 命名是否符合规范？
- [ ] 注释是否解释了"为什么"？
- [ ] 类型是否严格（无 any）？
- [ ] 导入顺序是否正确？
- [ ] 是否有冗余代码？

## 潜在风险

- [ ] 是否有竞态条件？
- [ ] 是否有内存泄漏风险？
- [ ] 是否有数据丢失风险？
- [ ] 是否有性能问题？

## 安全审计

- [ ] 是否存在 XSS 风险（innerHTML、unsafeHTML、用户输入未转义）？
- [ ] 是否存在原型污染风险（deep merge、Object spread 不可信数据）？
- [ ] 敏感信息是否可能被意外暴露（URL/token/key 硬编码、日志输出）？
- [ ] 第三方依赖是否有已知安全漏洞？

## 重构特定

- [ ] 功能清单是否完整？
- [ ] 删除功能是否标记？
- [ ] 测试是否更新？
- [ ] 文档是否更新？

## Review 流程

1. **Agent 自检**：修改代码后，按此清单自查
2. **独立 Review**：由另一个 agent 审核（见下方独立性约束）
3. **人工确认**：最终由人工确认后 commit

### 实施约束

- **不自动 commit**：实施阶段所有修改先放工作区并保持 unstaged，不自动暂存或提交，等人工确认后再处理 commit

### Review Agent 独立性约束

- Review Agent **不得参与**代码实施，只负责审核
- Review Agent **只看最终代码**，不看实施过程细节
- Review Agent **不受实施 Agent 的影响**，独立对照功能清单和测试覆盖进行检查

## Review 结果

- ✅ 通过：所有检查项通过
- ⚠️ 警告：有非阻塞性问题，可选择忽略
- ❌ 失败：有阻塞性问题，必须修复
