# 测试规范

## 最低测试要求

- 每个公共函数/方法必须有测试
- 必须覆盖：正常路径、边界条件、错误处理
- 降级场景必须有测试

## 边界用例清单

- 空输入（null, undefined, 空数组/对象）
- 最大/最小值（数值边界、字符串长度）
- 并发调用（异步函数的竞态条件）
- 网络异常（超时、断网、服务器错误）
- 存储不可用（IndexedDB/localStorage 不可用时的降级）

## 测试风格

- 测试描述使用中文
- 使用 `vi.fn<Type>()` 严格类型
- 避免 `as any`，使用具体类型
- Mock 函数需要类型参数
- 等待异步操作使用 `await new Promise(process.nextTick)` 而非 `setTimeout(50)`
- 只有需要验证调用（`toHaveBeenCalledWith`/`toHaveBeenCalled`）时才用 `vi.fn`，否则用普通函数

## 测试原则

- **AAA 模式**：Arrange（准备）→ Act（执行）→ Assert（断言），三段分明，空行分隔
- **一个测试只验证一个行为**：不把多个不相关的断言塞进一个测试
- **测试公共接口**：通过公共 API 验证行为，不测实现细节，重构时测试不应改变
- **测试间相互独立**：不依赖执行顺序，不共享可变状态，每个测试可单独运行

## 重构时的测试要求

- 重构前：记录现有测试用例清单
- 重构后：所有现有测试必须通过
- 新增功能：必须有对应测试
- 删除功能：必须人工确认并更新测试

## 功能清单模板（重构 Phase 1）

重构前必须产出功能清单，格式如下：

```markdown
## 功能清单

| 功能     | 描述                    | 状态 | 备注   |
| -------- | ----------------------- | ---- | ------ |
| 核心发送 | sendBeacon + fetch 降级 | 保留 | -      |
| 离线队列 | 离线时入队，在线时发送  | 保留 | -      |
| 失败重试 | 失败时入队，成功后重试  | 保留 | -      |
| 自动恢复 | 上线时自动 flush        | 删除 | 需确认 |
```

同时产出测试覆盖分析：已覆盖 / 缺失两部分。
